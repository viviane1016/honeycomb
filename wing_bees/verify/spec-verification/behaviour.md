## Behaviour

The verify subprocess is a fresh `claude -p` call on the same scribe model that authored the spec. Inputs: the spec text (as written at stage-spec) and the builder's git diff from the cell branch base (the point where the cell's branch was created) to the cell branch tip (the builder's final commit). The verify pass runs after the builder commits and before bees fast-forward-merges the cell branch into `feat/<slug>`. On APPROVE, dispatch proceeds directly to ff-merge. On first REJECT, a remediation loop revises the spec and re-dispatches the builder for a second attempt; on APPROVE of the second attempt, ff-merge proceeds. Only a second REJECT (or revision failure) hard-blocks ff-merge for that spec. Infrastructure errors (scribe timeouts, internal failures) do not block ff-merge (advisory, not blocking). Sidecars live at `.bees/<slug>/verify/NNN-attempt-<N>.md` (persisted post-cleanup) and are mirrored to `.bees/<slug>/cells/NNN/.verify-feedback.md` while the cell exists.
