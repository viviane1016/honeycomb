## Tone and clarity

- Be specific. "Fix the auth service" is vague; "Add OAuth 2.0 integration for GitHub and Google" is clear.
- Avoid jargon unless necessary. If you use acronyms (OAuth, JWT, SSO), assume the queen knows them; if unsure, add a one-word clarification.
- Distinguish "nice to have" from "must have". The `## In scope` and `## Out of scope` sections make this explicit.
- One briefing = one feature or fix. If you're juggling multiple independent changes, write multiple briefings and dispatch them separately.
