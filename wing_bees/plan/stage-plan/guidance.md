## Guidance

**Slug naming.** See `slug-naming` for naming rules, type prefixes, and examples.

**Briefing structure.** Write your briefing as if a code reviewer will read it in the PR. State the goal (what you want to build), the motivation (why it matters), the expected outcome (what success looks like), and any issue references. The briefing becomes the PR body's context; clarity here avoids misunderstandings later.

**Plan structure expectations.** Expect the plan to include `## Context`, `## Work breakdown` (with per-item builder-model recommendations), `## Risks`, `## Delivery estimates`, and `## Linked issues`. The work breakdown items become builder-sized specs; if an item seems too large, the queen will break it down further.
