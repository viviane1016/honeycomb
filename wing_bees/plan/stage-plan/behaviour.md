## Behaviour

The queen reads the operator's briefing, capped at 64 KiB. She surveys non-bees commits since the last feature merge to assess codebase drift, optionally recommending a scout run if changes are substantial. She then produces a structured plan listing work items, delivery estimates per item (with model recommendations), identified risks, and any linked issues extracted from the briefing. Output lives at `.bees/<slug>/plan.md`.

Common failure: the briefing is too vague, and the queen requests clarification rather than producing a half-built plan. Adversarial content in the briefing is caught by security scanning. Secret scan applies at entry (briefing, before the queen runs) and output (plan.md, after queen output, before commit) gates; see `wing_practices/secret-handling`.

Pre-flight tasks: if the queen-file has open petitions from an older honeycomb version, the queen reviews them first — pruning those now covered by upstream changes and version-rolling those still uncovered.

**Approval flow.** After the plan is written and committed to `feat/<slug>`, the default flow prompts `Approve and chain spec + dispatch? [y/N/edit]` on stdin. `y` chains spec and dispatch; `edit` opens `$EDITOR` on `plan.md` and re-prompts after save; `N`/Ctrl-C exits cleanly with the plan committed but no chained work. `--approve` skips the prompt; `--auto` skips the prompt and is **required when stdin is not a TTY** (e.g. `nohup … & disown` harness-detached runs). Any non-zero exit from a chained stage halts the chain with a `bees plan: chain halted at <stage>; resume with bees <stage> <slug>` message.

**Release map consultation.** If `RELEASE_MAP.md` exists at the repo root, the queen reads it during planning to understand how the requested feature fits the upcoming release sequence — what's queued for the next release, what's been deferred, what dependencies are in flight. The release map is maintained autonomously by the cartographer drone (see `.github/workflows/cartographer.yml`) and reflects the committed forward plan as of the last release. If `RELEASE_MAP.md` is absent (older repo or pre-cartographer state), this step is skipped. The queen may note in `## Context` that the feature naturally belongs to a specific release (e.g., "fits 1.2 Drone milestone alongside drone-sentinel"), but the operator's briefing remains authoritative.

**Depends-on mandate.** `QUEEN_PLAN_SYSTEM` instructs the queen: for any two specs in the work breakdown that touch the same file, the plan must declare an explicit `depends-on` relationship ordering them. Foundational refactor specs (cross-module type changes, central-module rewrites, state-shape changes) must be flagged with explicit dependents. Plans that omit `depends-on` for file-overlapping specs will produce wave-internal FF-merge failures that are expensive to diagnose; catching the gap at plan time costs nothing.
