## Behaviour

The ship stage begins with a queen-accept gate: a queen-bee invocation inside `ship.cmd_ship` that reviews the annotated `briefing.md` (including `## Operator amendments`), the plan (especially `## Acceptance`), all spec bodies, and the full feature diff (`git diff feat/<slug>...<base>`). The queen emits either `APPROVE` (with a one-line summary, which allows ship to proceed) or `CONCERNS` (a numbered list, written verbatim to `.bees/<slug>/accept.md` and printed to stderr). Queen-accept is advisory—the operator decides whether to address the concerns or override and ship. This is not a third human gate; the two human gates (plan approval and PR review) remain as defined in `human-gates`. The CONCERNS body is appended to the PR body by `build_pr_body` as an `## Acceptance concerns` section so the reviewer sees the queen's assessment.

After APPROVE, bees pushes the feature branch to origin and opens a PR using `gh pr create`. The PR body is assembled from the plan's `## Context` section, links to each spec file, and any issue references extracted from the briefing. The PR title is auto-derived from the plan's work breakdown. If the repo has CI configured (pytest, bandit, secret-scan), those checks run after the PR opens. The PR is the final human gate before merge.

**Pytest gate worktree isolation.** `bees ship` runs the pytest gate inside a temporary `git worktree add --detach <tmp> feat/<slug>` worktree, not in the primary checkout. The temp worktree is torn down in a try/finally block regardless of whether pytest passes, fails, or is interrupted. This ensures (a) the gate tests the exact code on the feature branch, not whatever happens to be in the primary working tree, and (b) no temp worktree persists after ship completes. If the feat branch has broken tests that the primary checkout does not, the gate will catch them. Bypass with `--skip-tests` if a known-broken test suite should not block ship.

Common failure: push rejected by branch protection rules or authentication issues. Surfaced directly to the user.

**Reconciliation step.** Immediately after a successful `push_branch`, `bees ship` runs `_reconcile_stranded_dispatch_events`. This helper reads `~/.bees/events.log` for the slug, identifies any spec NNN whose latest dispatch event is not `outcome ∈ {success, manual}` (i.e., a "stranded" spec), then checks whether a commit carrying the standard `Implements .bees/<slug>/specs/NNN-` footer exists on the feature branch (reusing the same git-log pattern as `already_merged_nnns` in `dispatch.py`). For each stranded NNN with a matching commit, it records the most-recent matching SHA and appends a `stage="dispatch", outcome="success", source="operator-repair"` event to `~/.bees/events.log`. The reconciler is idempotent: once the repair event is appended, the spec's latest dispatch event is `outcome="success"`, so a re-run of `bees ship` skips it. The reconciliation runs pre-PR so the PR body and dashboard already reflect the repaired state.
