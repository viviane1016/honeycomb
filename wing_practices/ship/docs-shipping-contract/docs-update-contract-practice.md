## Docs-update contract practice

The enforcement: before a feature lands, the scribe reviews the plan and adds a docs-update spec as the final work-breakdown item if any item changes something a user would notice. "Noticed" includes: (1) a new CLI flag or subcommand, (2) a change to the workflow (e.g. a new stage, a new human gate), (3) a change to the builder tool allowlist, (4) claims in the docs that the change falsifies. The builder bee who runs the docs-update spec edits whichever files the change affects: if it's a new CLI flag, the spec touches `--help` and `README.md`; if it's a workflow change, it touches `CLAUDE.md` and the relevant stage docs. The same spec also updates `BACKLOG.md` to mark items shipped and record deferred work. This ensures that no feature ships with outdated docs—the PR description will be confusing if the code does X but the README says it does Y.
