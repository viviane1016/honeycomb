## Observability practice

The canonical shape: each significant action (a plan starting, a builder deploying, an API request succeeding or failing) emits one JSONL record. The record carries the timestamp, the logical identity (which feature, which stage, which request ID), the outcome (success, timeout, 500 error), and the runtime context (model used, tokens consumed, duration). Exceptions during event emission are caught and swallowed—if the pipeline waited for observability to succeed, an observability bug would crash the whole system. Operators query the log with simple tools (`tail -f | grep`, `jq` filtering); that query becomes the incident-response workflow. "Why did deploy #123 hang?" → tail the log, grep by feature ID, find the wall-clock time where output stopped.
