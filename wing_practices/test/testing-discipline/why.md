## Why

Teams have been bitten repeatedly by the mock/prod divergence: a service's API changes, the mock doesn't, tests pass, the code ships, and then the real system rejects the request. Mocks are convenient but they're also a lie—they're your best guess at what the real system does, and guesses decay. The cost of running a real database, hitting a real API, or reading a real file is worth paying to avoid the cost of a production incident. Bees' own test suite refuses to mock the subprocess stages: the pipeline is tested end-to-end via actual `bees plan` / `bees spec` invocations, so the tests are slow but they're true. The V-model formalises this requirement-author-verifies principle: the role that authors requirements at a given level is the role that verifies them, so the test author verifies test outcomes and the plan author verifies plan delivery.
