## The antipattern

The shape: a client (browser, CLI, mobile app) needs to know when a long-running job completes. The naive implementation: client polls the server every N seconds asking "is it done?", expecting a response. This works but has costs: (1) **Cache overhead.** HTTP response caching is built on the assumption that GET requests fetch fresh or stale data based on cache headers. Polling violates that assumption—you're asking for fresh status every 5 seconds, so the response is never cached, and each request hits the database. (2) **Thread exhaustion.** Each polling client holds a server thread (or connection slot). If 100 clients poll every 5 seconds, the server is blocked responding to status requests instead of doing useful work. (3) **Rate limits.** APIs have request-rate limits. If you poll frequently, you hit them and lose updates (backoff, queue delays). (4) **Latency.** Updates arrive at poll intervals (if you poll every 5 seconds, you might wait 5 seconds to learn the job finished).
