## Why

Bees' dashboard initially polled `~/.bees/events.log` mtime every 5 seconds. This burned the query cache (mtime checks are cheap but non-free), tied up the single server thread, and delayed event display by up to 5 seconds. The fix: implement `/api/events/stream` using Server-Sent Events. Now the client opens one persistent connection; the server pushes updates as soon as they arrive; there's zero polling overhead. No cache is wasted, no request storms, no latency. The lesson: polling is always a workaround. The right tools are (1) SSE (server pushes updates over HTTP, works through proxies, low latency, minimal overhead), (2) WebSockets (bidirectional, lower latency, requires different protocol), (3) native push (mobile platforms), (4) webhooks (third-party integration, server pushes to your URL). Pick the tool that matches the constraint; polling should never be the first choice.
