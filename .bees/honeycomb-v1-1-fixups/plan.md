# Plan ‚Äî honeycomb v1.1 review fixups

## Context

`bees accept` on `feat/honeycomb-v1-1` (PR #4) flagged three concerns and three observations. The load-bearing one is that overlay precedence ‚Äî the central motivation of ADR-0002 ("submit a petition, recall it back immediately") ‚Äî is **broken end-to-end through the MCP wire**: `lib/honeycomb/recall.py` and `lib/honeycomb/semantic.py` accept `overlay_root=`, are unit-tested with it, but `bin/honeycomb-mcp` never threads the parameter into either dispatch call. The petition tools (`palace_petition_submit`, `palace_petition_list`) already use a local `_overlay_root_for_petitions()` helper, so the resolution rule exists ‚Äî it's just not applied at recall dispatch.

The other items are smaller but visible: `tools/install.sh` writes `.petition-manifest.json` and prints a `Petitions:` line on every install (breaks the "byte-identical-to-v1.0 for a no-change canon" acceptance bullet); two adjacent install.sh sections are both labelled `# ‚îÄ‚îÄ 4.` (cosmetic); the `petition_id` `YYYYMMDD-NNN-<scope>` scheme is over-engineering noted in retro (the override file's path *is* the identity per ADR-0002); `_load_log_writer` in `bin/honeycomb-mcp` re-imports per call and accumulates `sys.path` entries; and the drawer order in `wing_bees/plan/palace-petitions/index.md` doesn't match the order spec 010 narrated.

This fixup ships on top of `feat/honeycomb-v1-1` (or directly to `main` if PR #4 has merged by the time the colony dispatches). It is purely a tightening pass ‚Äî no new architectural decisions, no API additions beyond a single new result-dict field (`source`).

## Approach

Four units, batched into three dependency waves:

1. **MCP server tweaks** (Wave 1, parallel-safe with Unit 2): wire `overlay_root` through the recall + semantic dispatches in `bin/honeycomb-mcp`, generalise the existing petition-only `_overlay_root_for_petitions()` helper into a single `_overlay_root()` used by all four tool paths, add a `source` field (`"canon"` or `"consumer-overlay"`) to result dicts in `recall.py` / `semantic.py` so callers can distinguish, and cache `_load_log_writer` at module level so `sys.path.insert` and `honeycomb.log` import happen exactly once. Co-scoped because all four sub-changes live in `bin/honeycomb-mcp` (plus the two lib files for the `source` field), share a single PR review, and would otherwise produce three trivial specs against the same file.

2. **install.sh manifest gate + renumber** (Wave 1, parallel-safe with Unit 1): wrap the petition-manifest block so it (a) skips the `Petitions:` summary print and the manifest write entirely when `PREVIOUS_SHA == CURRENT_SHA` (no-op install ‚Äî byte-identical to v1.0) and (b) on a fresh install writes the manifest with `previous_sha=null` but does not print the summary line. Renumber the second `# ‚îÄ‚îÄ 4.` to `# ‚îÄ‚îÄ 5.`.

3. **Drop `petition_id` from the petition API** (Wave 2, depends on Unit 1 for the shared file lock): remove the field from `PetitionResult`, `PendingPetition`, the queenfile drawer's HTML-comment frontmatter, and the MCP descriptor docs for `palace_petition_submit` / `palace_petition_list`. Rewrite `palace_petition_submit` to compute its branch deterministically without the date-and-count scheme, and rewrite `palace_petition_withdraw` to take the override file's relative path (its identity per ADR-0002) instead of `petition_id`, recomputing the branch from path. Update the two affected test files.

4. **Docs sync + index reorder + BACKLOG sweep** (Wave 3, depends on Units 2 and 3 for accurate description of landed state): scribe-only unit that updates ADR-0002 with a supersession note ("petition identity is path, not id"), rewrites the `petition_id`-mentioning drawers in `wing_bees/plan/palace-petitions/` and `wing_bees/plan/petitions-flow/`, reorders `palace-petitions/index.md` to match spec 010's narrative (migration last), trims README.md's install-summary claim, and records the fixup ship in BACKLOG.md and CHANGELOG.md.

## Acceptance

- All pre-existing tests pass: `python3 -m pytest tests/`.
- `tests/test_mcp_overlay_e2e.py` passes: a petition submitted through the MCP server is recallable through the same MCP server in the same session, with the result entry carrying `source == "consumer-overlay"` and the overlay body content.
- `tests/test_install_no_op.py` passes: running `bash tools/install.sh` twice against an unchanged local-fixture canon produces a byte-identical install root after the second run and no `Petitions:` stdout from the second run.
- `tests/test_log_writer_cache.py` passes: `_load_log_writer` performs the `honeycomb.log` import exactly once across multiple invocations.
- `grep -nE "petition_id" lib/honeycomb/ bin/honeycomb-mcp` returns no matches (test fixtures may legitimately retain the string if they test legacy-content tolerance).
- `tools/install.sh` section comments are monotonically increasing: `# ‚îÄ‚îÄ 1.`, `# ‚îÄ‚îÄ 2.`, `# ‚îÄ‚îÄ 3.`, `# ‚îÄ‚îÄ 4.`, `# ‚îÄ‚îÄ 5.` each unique.
- `wing_bees/plan/palace-petitions/index.md` drawer order ends with `migration`.
- Submitting `palace_petition_submit` via MCP returns a JSON object whose keys are `{branch, pr_url, overlay_path}` ‚Äî no `petition_id`. Querying `palace_petition_list` returns entries whose shape is `{target, consumer, tool, tool_version, path, source, rationale}` (or equivalent) ‚Äî no `petition_id`. Reading any generated `.queenfile_*.md` HTML comment shows the documented fields (`target`, `tool`, `tool_version`, `consumer`, `rationale`) ‚Äî no `petition_id` line.
- `palace_petition_withdraw` invoked with the override file's path closes the PR and removes the branch as before.
- ADR-0002 carries a supersession note recording the petition-identity change. CHANGELOG and BACKLOG reflect the fixup ship.

## Risks / open questions

- **Withdraw identifier shape.** Plan specifies `palace_petition_withdraw(path: str)`. An alternative is `palace_petition_withdraw(branch: str)` since `submit` already returns `branch`. The scribe should pick one and apply consistently across MCP descriptor, library API, tests, and `petitions-flow/lifecycle.md`. Path-as-identity is preferred since it matches ADR-0002's framing; branch-as-key is acceptable if the scribe finds the derivation logic clearer to read.
- **Branch derivation from path.** The new deterministic branch name needs to be unique across all (wing, room, closet, drawer, scope) tuples. Scribe to pick either a path-slug form (`feat/petition-<wing>-<room>-<closet>-<filename-stem>`, verbose but readable) or a stable 12-char hash (`feat/petition-<sha1(rel_path)[:12]>`, opaque but uniform-length). Both submit and withdraw must use the same function.
- **install.sh testability.** The new `tests/test_install_no_op.py` requires `tools/install.sh` to accept a `HONEYCOMB_REPO` env override pointing at a local fixture (so the test can avoid the real GitHub URL). The plan includes the install.sh edit. If the spec writer judges the local-fixture clone too fragile to test reliably under CI, fall back to: drop the integration test and replace with a `tests/test_manifest.py` extension that asserts `summary_line` returns empty for the no-op case plus a shell-script-level grep that the gate condition exists in install.sh.
- **VERSION/CHANGELOG bump.** Whether this fixup amends v1.1.0 or becomes v1.1.1 depends on whether v1.1.0 has been tagged on `origin/main` at the time the docs unit runs. The scribe checks `git tag --list 'v1.1.*'` and picks accordingly. No operator gate is needed mid-flight; the scribe encodes the conditional in the spec body.
- **Existing `.queenfile_*.md` files in the wild.** If any consumer has already installed v1.1.0 and submitted a petition, those override files have `petition_id:` in their HTML comment. The parser already ignores unknown fields (it only reads `target`, `tool`, `tool_version`, `consumer`, `rationale`), so backwards-compat is automatic. No migration script needed; the fixup does not need a deprecation window.
- **`source` field value canonicalisation.** Plan specifies `"canon"` and `"consumer-overlay"`. The e2e test asserts the literal string `"consumer-overlay"`. Spec writer should define these as module-level constants in `lib/honeycomb/recall.py` (e.g. `SOURCE_CANON = "canon"`, `SOURCE_OVERLAY = "consumer-overlay"`) so the test imports them rather than hardcoding string literals.
